# 初学者完整调试流程

这份文档按“从最简单到最复杂”的顺序写。不要一开始就把所有线都接上，也不要一上来就让小车落地跑。每一步只验证一个东西，确认没问题再进入下一步。

## 0. 现在的引脚是否够用

够用。当前工程的关键引脚分配如下：

| 模块 | 引脚 |
| --- | --- |
| 板载 LED | `PC13` |
| PS2 | `PB12`、`PB13`、`PB14`、`PB15` |
| OLED | `PB8=SCL`、`PB9=SDA` |
| 四路 PWM | `PA8`、`PA9`、`PA10`、`PA11` |
| TB6612 方向 | `PB0`、`PB1`、`PB10`、`PB11`、`PA12`、`PA15`、`PB4`、`PB5` |
| 编码器 | `PA0`、`PA1`、`PA6`、`PA7`、`PB6`、`PB7`、`PA4`、`PA5` |
| 下载调试 | `PA13=SWDIO`、`PA14=SWCLK` |

因为用了 `PB4`，程序会关闭 JTAG、保留 SWD。也就是说 ST-Link 的 `SWDIO/SWCLK` 下载不受影响。

## 0.1 电机和编码器参数

当前使用的是 12V 333rpm 的 520 霍尔编码器减速电机，减速比为 30，编码器基础脉冲为 11PPR。

计算方式：

```text
输出轴基础脉冲 = 11PPR * 30 = 330PPR
程序四倍频计数 = 330 * 4 = 1320 count/rev
333rpm = 333 / 60 = 5.55 rev/s
最大计数速度 = 5.55 * 1320 = 7326 count/s
10ms 内最大计数 = 7326 * 0.01 = 73 count
```

也就是说，OLED 上看到的单轮编码器增量，正常情况下不会动不动几百。之前如果把目标速度设成几百，电机根本追不上，PID 会一直输出很大，车会很难控制。

当前代码已经使用保守起步值：

```c
#define SPEED_LIMIT_TICKS 35
#define SLOW_SPEED_TICKS  18
#define PID_KP            12
#define PID_KI            1
#define PID_KD            0
```

如果完全没经验，第一次落地还可以改得更慢：

```c
#define SPEED_LIMIT_TICKS 20
#define SLOW_SPEED_TICKS  10
```

## 1. 只拿最小系统板测试 LED

### 目标

确认最小系统板、Keil 工程、下载器、系统时钟、`SysTick`、GPIO 输出都正常。

### 接线

只需要接：

- ST-Link `3V3` 到板子 `3V3`
- ST-Link `GND` 到板子 `GND`
- ST-Link `SWDIO` 到 `PA13`
- ST-Link `SWCLK` 到 `PA14`

如果你的 ST-Link 不给板子供电，就用外部 3.3V 给最小系统板供电，但必须共地。

### 现象

下载程序后，`PC13` 板载 LED 应该每 500ms 左右闪一次。

### 主函数关键代码解释

代码位置：`Core/main.c`

```c
gpio_clock_enable();
```

打开 GPIO 和 AFIO 的外设时钟。STM32 的外设默认是关闭时钟的，如果不先打开，后面配置引脚不会生效。

```c
systick_init();
```

启动 1ms 系统节拍。后面的 LED 闪烁、10ms 控制循环、OLED 刷新都靠 `millis()` 计时。

```c
led_init();
```

把 `PC13` 配置成输出。常见 C8T6 最小系统板的 LED 是低电平点亮，所以程序里会反着控制。

```c
if ((millis() - last_led_tick) >= LED_HEARTBEAT_MS) {
    last_led_tick += LED_HEARTBEAT_MS;
    led_toggle();
}
```

这几行负责让 LED 周期闪烁。只要主循环还在运行，LED 就会一直闪。如果 LED 不闪，先不要接电机，先检查下载、供电和 PC13 LED。

### 如果 LED 不闪

按顺序检查：

1. Keil 是否真的下载成功。
2. ST-Link 接线是否正确：`PA13/PA14/GND/3V3`。
3. 板子电源灯是否亮。
4. 你的最小系统板 LED 是否确实接在 `PC13`。
5. Keil 是否选择了 `STM32F103C8` 目标芯片。

## 2. 接 OLED，确认显示正常

### 目标

OLED 用来显示 PS2 状态、使能状态和四个编码器读数。初学者调车时很有用，因为不用一开始就上串口。

### 接线

| OLED | STM32 |
| --- | --- |
| `VCC` | `3V3` |
| `GND` | `GND` |
| `SCL` | `PB8` |
| `SDA` | `PB9` |

### 现象

上电后 OLED 会显示类似：

```text
PS2:WAIT EN:0
FL:0 FR:0
RL:0 RR:0
START GO SELECT STOP
```

如果 OLED 不亮：

1. 检查 VCC/GND 有没有接反。
2. 确认 OLED 是 I2C 版本，不是 SPI 版本。
3. 常见地址是 `0x78`。如果你的 OLED 地址不同，需要改 `Drivers/oled.c` 里的 `OLED_ADDR`。
4. OLED 不亮不会影响 LED 和电机程序运行。

## 3. 测试 PS2 手柄

### 目标

确认 PS2 接收器能被 STM32 读到。

### 接线

| PS2 接收器 | STM32 |
| --- | --- |
| `VCC` | `3V3` |
| `GND` | `GND` |
| `ATT` / `CS` | `PB12` |
| `CLK` | `PB13` |
| `DAT` | `PB14` |
| `CMD` | `PB15` |

### 现象

PS2 正常时，OLED 第一行应该从：

```text
PS2:WAIT
```

变成：

```text
PS2:OK
```

按 `START` 后：

```text
EN:1
```

按 `SELECT` 后：

```text
EN:0
```

### 如果 PS2 不正常

1. 先确认接收器供电是 `3V3`。
2. 检查 `DAT/CMD/CLK/ATT` 有没有接反。
3. 手柄和接收器是否已经配对。
4. 有些 PS2 接收器需要先按手柄上的模式键进入模拟模式。
5. 如果 OLED 一直 `WAIT`，先不要接电机。

## 4. 测试电机方向

### 目标

确认四个电机在正命令下方向一致，并符合麦轮安装方向。

### 安全要求

车必须架起来，四个轮子离地。第一次不要落地。

### 接线

按 `docs/wiring.md` 连接网上的 TB6612 四路电机驱动模块。后右电机方向已经改为：

| 后右轮 | 引脚 |
| --- | --- |
| PWM | `PA11` |
| IN1 | `PB4` |
| IN2 | `PB5` |

### 测试方法

1. 按住 `L1`，让车处于慢速模式。
2. 按 `START` 使能。
3. 轻推左摇杆向前。
4. 观察四个轮子是否都让车“理论上向前走”。

如果某个轮子方向反了：

1. 优先对调这个电机接 TB6612 输出的两根线。
2. 再测试一次。
3. 不要一开始就改代码，因为线接反是最常见问题。

## 5. 检查编码器读数

### 目标

确认每个轮子的编码器能读到数，并且方向和电机方向一致。

### OLED 显示

OLED 会显示：

```text
FL:xxx FR:xxx
RL:xxx RR:xxx
```

这些数值是每个 10ms 控制周期内的编码器增量，不是 RPM，也不是总里程。对当前 520 电机来说，空载最大也大约是 73 count/10ms，所以调试时看到几十是正常的，看到几百通常表示参数理解错了、计数方向乱了或噪声很大。

如果刚上电时看到 `FL:-12287`、`FR:8193` 这种上万的数字，它没有特殊含义，属于异常值。常见原因是编码器线没接、A/B 相悬空、有强干扰，或者刚启动时旧计数基准没有清掉。当前程序已经在初始化和手柄掉线时调用 `encoder_reset()` 清掉基准；如果仍然出现上万，优先检查编码器接线和地线共地。

判断大致范围：

| OLED 数值 | 含义 |
| --- | --- |
| `0` | 轮子没动，或编码器没有读到变化 |
| `1-10` | 很慢地转 |
| `10-40` | 适合初学者低速调试 |
| `40-80` | 接近较快速度或空载高速 |
| `几百以上` | 大概率异常，不要直接闭环落地跑 |

### 测试方法

1. 车架起来。
2. 不按 `START`，用手慢慢转某个轮子。
3. 看对应轮子的数值是否变化。
4. 只转前左轮时，应该主要是 `FL` 变化；只转前右轮时，应该主要是 `FR` 变化。

如果读数不变：

1. 检查编码器 VCC/GND。
2. 检查 A/B 相是否接到对应引脚。
3. 确认编码器输出电平对 STM32 安全。

如果电机正转时编码器读数是负的：

1. 对调这个轮子的编码器 A/B 相。
2. 或者在代码里对这个轮子的读数取反。
3. 对初学者来说，优先对调 A/B 线更直观。

## 6. PID 调试

### 目标

让每个轮子的实际速度跟上目标速度，并且不明显抖动。

### 初始建议

先在 `Core/config.h` 里使用保守参数：

```c
#define SPEED_LIMIT_TICKS 35
#define SLOW_SPEED_TICKS  18
#define PID_KP            12
#define PID_KI            1
#define PID_KD            0
```

如果控制不好，可以继续降低：

```c
#define SPEED_LIMIT_TICKS 20
#define SLOW_SPEED_TICKS  10
```

### 调试顺序

1. 先把车架起来。
2. 把 `PID_KI` 改成 0，`PID_KD` 保持 0。
3. 只调 `PID_KP`。
4. 如果轮子反应很慢，稍微增大 `PID_KP`。
5. 如果轮子来回抖，减小 `PID_KP`。
6. 方向和速度都稳定后，再加一点 `PID_KI`。
7. `PID_KD` 暂时不要动。霍尔编码器读数颗粒感比较明显，D 项容易放大噪声。

### 判断 PID 是否有问题

| 现象 | 可能原因 |
| --- | --- |
| 一使能轮子猛冲 | 电机方向或编码器方向反了，或者 `PID_KP` 太大 |
| 轮子抖动 | `PID_KP` 太大，编码器读数噪声大 |
| 低速不转 | 摩擦大，`PID_KI` 太小，或者速度目标太低 |
| 越调越乱 | 先回到低速度、低 KP，只测一个方向 |

## 7. 落地测试

### 第一次落地

1. 找一块空地。
2. 继续按住 `L1` 慢速模式。
3. 只轻推一点点摇杆。
4. 先测前进/后退。
5. 再测左右平移。
6. 最后测自转。

### 摇杆限幅建议

程序已经把默认速度调低了：

```c
#define SPEED_LIMIT_TICKS 35
#define SLOW_SPEED_TICKS  18
```

如果对方是完全初学者，建议第一次落地改得更慢：

```c
#define SPEED_LIMIT_TICKS 20
#define SLOW_SPEED_TICKS  10
```

确认能控制住以后，再一点点加。

### 额外建议

- 第一次不要装太重的电池或外壳，先让底盘裸跑。
- 电机电源和 STM32 电源最好分清楚，但必须共地。
- 电机线和编码器线不要绑在一起走太长，减少干扰。
- 每次只改一个东西：只改一个 PID 参数，或者只换一个轮子的线。
- 改完要记录，不要靠记忆。
- 小车没有陀螺仪，跑久了方向会偏，这是正常的，不是代码一定错。
